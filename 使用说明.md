## 4. Skills 详细说明

### 4.1 `using-superpowers`

**用途**

这是一个流程型 skill。它的核心作用不是完成具体业务任务，而是要求模型在开始处理任务之前，优先判断“有没有合适的 skill 应该先被调用”。它适合拿来规范整个工作流，让 Codex 更稳定地利用已经安装的 skill 体系。

**适合场景**

- 你希望 Codex 在任何复杂任务开始前先检查有没有 skill 可用
- 你安装了很多 skill，希望模型优先走 skill 驱动流程
- 你想减少“明明有 skill 却没用上”的情况

**怎么使用**

通常不需要你每次都手动调用它。更常见的方式是让它作为一条“总规则”存在，让 Codex 在会话开始时优先检查 skill 适配性。

**使用效果**

- 提高 skill 命中率
- 让工作流更一致
- 更适合多 skill 环境

**注意事项**

- 它偏“流程控制”，不是具体功能型工具
- 如果用户有明确要求，仍然以用户要求为准

### 4.2 `playwright`

**用途**

用于浏览器自动化。它可以驱动真实浏览器完成打开网页、抓取页面、点击元素、输入内容、切换标签页、截图等动作。

**适合场景**

- 自动访问网站并截图
- 自动点按钮、填写表单
- 调试页面交互流程
- 抓取页面中的结构化信息

**前置条件**

- 系统需要有 `npx`
- 推荐有 Node.js 和 npm
- skill 默认通过 `playwright-cli` 或包装脚本运行

**推荐使用方法**

先设置 skill 脚本路径：

```bash
export CODEX_HOME="${CODEX_HOME:-$HOME/.codex}"
export PWCLI="$CODEX_HOME/skills/playwright/scripts/playwright_cli.sh"
```

然后执行：

```bash
"$PWCLI" open https://playwright.dev --headed
"$PWCLI" snapshot
"$PWCLI" click e15
"$PWCLI" type "Playwright"
"$PWCLI" press Enter
"$PWCLI" screenshot
```

**典型工作流**

1. 打开页面
2. 执行 `snapshot` 获取元素引用
3. 使用元素引用点击、输入、跳转
4. 页面变化后重新 `snapshot`
5. 最后截图或导出结果

**注意事项**

- 页面变动后旧元素引用可能失效
- 如果机器上没有 `npx`，这个 skill 不能直接工作

### 4.3 `markitdown`

**用途**

把各种文档和内容转换成 Markdown，适合给 LLM、知识库、Obsidian、文档归档流程使用。

**支持的输入**

- PDF
- DOCX
- PPTX
- XLSX
- 图片
- 音频
- HTML
- CSV
- JSON
- XML
- ZIP
- EPUB
- YouTube 链接

**适合场景**

- 把论文 PDF 转成 Markdown
- 把 Word、PPT、Excel 资料变成可搜索文本
- 把网页内容转成便于整理的 Markdown
- 为知识库做内容标准化

**命令行用法**

```bash
markitdown document.pdf > output.md
markitdown document.pdf -o output.md
markitdown --list-plugins
markitdown --use-plugins document.pdf -o output.md
```

**Python 用法**

```python
from markitdown import MarkItDown

md = MarkItDown()
result = md.convert("document.pdf")
print(result.text_content)
```

**适合你怎么用**

- 你拿到一个 PDF 或 DOCX，就可以先用它转成 Markdown
- 再把转出来的内容送进 Obsidian、总结流程或其他 AI 工作流

**注意事项**

- 某些高级功能依赖额外插件或模型接口
- 图片 OCR、音频转录、AI 描述这类增强功能通常需要额外环境

### 4.4 `obsidian-markdown`

**用途**

帮助你创建和编辑 Obsidian 风格 Markdown，而不是普通 Markdown。它特别适合处理 wikilinks、embeds、callouts 和 frontmatter。

**适合场景**

- 维护 Obsidian 笔记库
- 新建支持 `[[wikilink]]` 的笔记
- 编写带 callout 的学习笔记、项目笔记
- 编辑带 frontmatter 属性的笔记

**典型写法**

```markdown
---
title: My Note
tags:
  - project
  - idea
---

# 主题

相关笔记：[[Another Note]]

> [!note]
> 这是一个 Obsidian callout。

![[diagram.png]]
```

**使用方法**

你可以直接告诉 Codex：

```text
请使用 obsidian-markdown 帮我创建一篇 Obsidian 笔记
```

**注意事项**

- 它强调的是 Obsidian 扩展语法
- 外部链接仍然用普通 Markdown 链接

### 4.5 `obsidian-bases`

**用途**

用于创建和编辑 Obsidian 的 `.base` 文件，帮助你搭建数据库式视图，例如表格视图、卡片视图、列表视图和分组视图。

**适合场景**

- 用标签和属性筛选笔记
- 做任务数据库、项目视图、读书清单
- 在 Obsidian 里构建更结构化的展示层

**基础结构**

```yaml
filters:
  and:
    - 'status == "done"'

views:
  - type: table
    name: "Done Tasks"
    order:
      - file.name
      - status
```

**使用方法**

```text
请使用 obsidian-bases 帮我创建一个 Obsidian `.base` 文件
```

**注意事项**

- `.base` 文件本质是 YAML
- 最常见的问题是引号、公式引用和 YAML 缩进错误

### 4.6 `obsidian-cli`

**用途**

通过 Obsidian CLI 与正在运行的 Obsidian 交互，包括读写笔记、搜索内容、修改属性、管理任务，以及辅助插件开发调试。

**适合场景**

- 从命令行管理笔记库
- 自动创建和更新 Daily Note
- 搜索某个主题的所有笔记
- 调试 Obsidian 插件

**前置条件**

- Obsidian 需要处于运行状态
- 系统里需要有 `obsidian` CLI

**常见命令**

```bash
obsidian read file="My Note"
obsidian create name="New Note" content="# Hello"
obsidian append file="My Note" content="New line"
obsidian search query="project"
obsidian property:set name="status" value="done" file="My Note"
```

**插件开发常见命令**

```bash
obsidian plugin:reload id=my-plugin
obsidian dev:errors
obsidian dev:screenshot path=screenshot.png
```

**注意事项**

- 如果 Obsidian 没开，这个 skill 的很多操作不会成功

### 4.8 `defuddle`

**用途**

把网页提取成干净 Markdown，去掉站点导航、广告、侧边栏和杂乱元素，减少 token 浪费。

**适合场景**

- 阅读文档网站
- 保存博客文章
- 抓取说明页内容做归档
- 给 AI 做更干净的网页输入

**命令行用法**

```bash
defuddle parse <url> --md
defuddle parse <url> --md -o content.md
defuddle parse <url> -p title
```

**使用方法**

```text
请使用 defuddle 提取这个网页正文并保存成 Markdown
```

**注意事项**

- 如果 URL 本身就是 `.md` 文件，通常直接读取原 Markdown 更合适
- 依赖系统已经安装 `defuddle` CLI

### 4.10 `security-threat-model`

**用途**

这是目前这批 skill 里最适合用来做安全审查的一个。它会对仓库或目录进行威胁建模，分析信任边界、敏感资产、攻击路径、影响面和缓解方案。

**适合场景**

- 审查第三方 skill 是否值得安装
- 审查一个本地 skill 目录有没有明显风险
- 审查某个仓库的敏感入口和安全边界

**推荐使用方式**

```text
请使用 security-threat-model 审查 /Users/bobo/.codex/skills/markitdown
重点关注：
1. 是否会执行外部命令
2. 是否访问网络
3. 是否可能读写敏感目录
4. 给出高/中/低风险结论
```

**输出通常会包含**

- 系统模型
- 信任边界
- 关键资产
- 入口点
- 主要威胁路径
- 风险等级
- 缓解建议

**注意事项**

- 它不是杀毒软件
- 它更偏结构化 AppSec 分析

### 4.11 `pdf`

**用途**

用于读取、生成、审查 PDF，重点是“视觉和版式准确性”，而不是只做纯文本提取。

**适合场景**

- 程序化生成 PDF 报告
- 检查 PDF 页面渲染是否正常
- 提取 PDF 内容做辅助检查

**建议工作流**

1. 生成或读取 PDF
2. 把 PDF 页面渲染为 PNG 检查版式
3. 用 `pdfplumber` 或 `pypdf` 做文本提取
4. 再回头检查视觉质量

**依赖**

```bash
uv pip install reportlab pdfplumber pypdf
```

系统渲染工具：

```bash
brew install poppler
```

**注意事项**

- 只提文本不能代表排版没问题
- 真正交付前最好总是做一轮渲染预览

## 5. 推荐组合用法

下面是几种非常实用的 skill 组合：

### 文档整理流

- `defuddle`：提取网页正文
- `markitdown`：把文档转成 Markdown
- `obsidian-markdown`：整理成 Obsidian 笔记

### Obsidian 知识库流

- `obsidian-markdown`：写笔记
- `obsidian-bases`：搭建数据库视图
- `json-canvas`：画知识图谱
- `obsidian-cli`：自动化管理笔记

### 安全审查流

- `find-skills`：先找 skill
- `security-threat-model`：再评估是否安全

### 网页自动化流

- `playwright`：自动打开、点击、截图
- `defuddle`：提取网页正文内容
